# System Architecture Document (SAD): DECP Platform

---

## 1. System Overview

* **Purpose:** The DECP (Department Engagement & Career Platform) is a role-aware, full-stack social and professional networking platform built for university departments. It exposes a RESTful API surface via a single entry-point API Gateway that proxies requests to nine independent backend microservices, each owning its own isolated PostgreSQL database. Real-time communication is provided through a Socket.IO WebSocket layer on the Messaging Service. The system serves both a React SPA web client and a React Native / Expo mobile application.

* **Architecture Pattern:** **Microservices** (Service-Oriented Architecture). Each domain capability (authentication, user profiles, feed, jobs, events, research, messaging, notifications, analytics) is encapsulated in its own independently deployable Node.js service with a private database. Services communicate exclusively through the API Gateway — there is no direct service-to-service REST calling between domain services. The API Gateway performs path-based routing, JWT validation, identity injection, rate limiting, and multipart passthrough. The Messaging Service adds a **WebSocket / event-driven** layer (Socket.IO + Redis pub/sub) on top of REST for real-time delivery.

* **Primary Tech Stack:**

  | Layer | Technologies |
  |---|---|
  | **Web Frontend** | React 18, TypeScript 5, Vite 7, Redux Toolkit 2 + RTK Query, MUI v5, Tailwind CSS 3, Socket.IO Client 4, React Hook Form 7, Zod 3, Recharts 2 |
  | **Mobile Client** | React Native 0.73, Expo ~50, TypeScript 5, Redux Toolkit 2 + redux-persist, React Navigation v6, Socket.IO Client 4, React Native Paper 5, Expo Notifications, expo-secure-store |
  | **API Gateway** | Node.js 20, TypeScript 5, Express 4, http-proxy-middleware 2 / custom axios proxy, Helmet 7, express-rate-limit 7, jsonwebtoken 9, Winston 3 |
  | **Backend Services (×9)** | Node.js 20, TypeScript 5, Express 4, Sequelize 6 ORM, Socket.IO 4 (Messaging), bcryptjs, Nodemailer 6, web-push 3, Joi 17, Winston 3 |
  | **Primary Database** | PostgreSQL 16 (one isolated database per microservice, 9 total) |
  | **Caching / Pub-Sub** | Redis 7 (session-adjacent pub/sub for Socket.IO adapter; `allkeys-lru` eviction policy in production) |
  | **Infrastructure (Production)** | AWS EKS (Kubernetes), AWS RDS PostgreSQL, AWS ElastiCache Redis, AWS S3 + CloudFront, Terraform ≥1.5 IaC, GitHub Actions CI/CD |
  | **Observability** | Prometheus (scrape interval 15s), Grafana (dashboards), Loki (log aggregation), Alertmanager |

## 3. Component Architecture

### 3.1 Frontend / Client — Web Application

* **Framework:** React 18.2 with TypeScript 5.2, bundled by **Vite 7** (ESBuild + Rollup). The production artefact is a statically compiled SPA served by an **NGINX Alpine** Docker container. The NGINX config handles client-side routing (`try_files $uri /index.html`) and proxies `/api` and `/socket.io` upstream.

* **State Management:** **Redux Toolkit 2** (`configureStore`) with **RTK Query** (`createApi`) as the data-fetching and server-state caching layer. The RTK Query `apiSlice` defines nine tag types (`User`, `Post`, `Job`, `Event`, `Research`, `Chat`, `Message`, `Notification`, `Analytics`) and automatically handles cache invalidation. A `createListenerMiddleware` listener wires the `logout` action to flush all cached API state. A custom `baseQueryWithReauth` wrapper intercepts 401 responses, silently attempts a token refresh via `POST /api/v1/auth/refresh`, retries the original request with the new token, and dispatches `logout()` if the refresh fails — providing transparent token rotation to all UI consumers.

* **Key Libraries:**
  * `@mui/material` v5 + `@emotion/react` — component library and theming
  * `tailwindcss` 3 + `postcss` + `autoprefixer` — utility-class styling alongside MUI
  * `react-hook-form` 7 + `@hookform/resolvers` + `zod` 3 — form state management with schema validation

  * `react-router-dom` 6 — client-side routing
  * `react-dropzone` 14 — drag-and-drop file upload interface
  * `axios` 1 — used alongside RTK Query for non-slice requests

* **Testing:** Vitest 4 + `@testing-library/react` 14 + `jsdom` 24 (unit and component tests); `@vitest/coverage-v8` for coverage.

### 3.3 Backend / API Layer

#### API Gateway (Port 3000)

The API Gateway is the sole publicly-exposed backend process. It acts as a **reverse proxy** implemented entirely in Express 4 with a custom `createProxyHandler` factory function (rather than a third-party proxy library for the primary path, with `http-proxy-middleware` present as a dependency but replaced by an `axios`-based proxy and a native `http`/`https` streaming handler for multipart uploads).

**Request processing pipeline (in order):**
1. `helmet` — sets 11 security HTTP response headers
2. `cors` — validates `Origin` header against allowlist; allows all origins in development
3. `compression` — gzip response compression
4. `express.json({ limit: '10mb' })` — body parsing
5. `morgan` — HTTP access logging through `winston`
6. `express-rate-limit` (global) — 100 requests per 15-minute window per IP
7. `correlationIdMiddleware` — generates or propagates an `X-Correlation-ID` UUID for distributed tracing
8. Route-level `authMiddleware` — verifies `Authorization: Bearer <jwt>` using `jsonwebtoken`; populates `req.user`; injects `x-user-id`, `x-user-role`, `x-user-email` headers into the forwarded request
9. Route-level `strictRateLimiter` — 5 requests per 15 minutes (applied only to `/auth/register`, `/auth/login`, `/auth/forgot-password`, `/auth/reset-password`)
10. `createProxyHandler(targetServiceURL, servicePrefix?)` — strips the `/api/v1` and optional service-name prefix from the URL path, adds `x-internal-token` header (SEC-002 internal trust token), and forwards via `axios` (JSON/form) or native `http.request` streaming pipe (multipart/form-data)

**Service URL map (from `config.services`):**

| Service | Internal URL (Docker) | Port |
|---|---|---|
| auth-service | `http://auth-service:3001` | 3001 |
| user-service | `http://user-service:3002` | 3002 |
| feed-service | `http://feed-service:3003` | 3003 |
| jobs-service | `http://jobs-service:3004` | 3004 |
| events-service | `http://events-service:3005` | 3005 |
| research-service | `http://research-service:3006` | 3006 |
| messaging-service | `http://messaging-service:3007` | 3007 |
| notification-service | `http://notification-service:3008` | 3008 |
| analytics-service | `http://analytics-service:3009` | 3009 |

* **API Design:** RESTful. All endpoints are versioned under `/api/v1/`. Resources follow noun-based URL conventions (`/api/v1/jobs`, `/api/v1/events/:eventId/rsvp`). Responses follow a consistent envelope: `{ success: boolean, message: string, data: T }`.

* **Authentication:** Stateless JWT-based authentication using the RS256-compatible HS256 signing algorithm (`jsonwebtoken` 9). The gateway is the sole JWT verification point — downstream services trust the `x-user-id` and `x-user-role` headers injected by the gateway rather than re-verifying the token. This is hardened by the `x-internal-token` header: each downstream service's `internalAuthMiddleware` rejects requests that do not carry this shared secret, preventing direct access bypassing the gateway.

#### Auth Service (Port 3001)

Built with `passport` 0.7 + `passport-google-oauth20` and `passport-jwt`. Key capabilities:
* **Local registration:** Joi-validated email + password (`bcryptjs` salt rounds 10), UUID v4 email verification token, Nodemailer SMTP dispatch.
* **Google OAuth 2.0:** `passport-google-oauth20` strategy; on first login creates a user record with `isEmailVerified: true` and no password hash.
* **JWT issuance:** `jsonwebtoken.sign()` with 15-minute access token + 7-day refresh token. Refresh tokens are persisted in the `refresh_tokens` table (Sequelize model `RefreshToken`) to enable server-side revocation.
* **Password reset:** UUID v4 reset token with `passwordResetExpires` timestamp stored on the `users` table.
* **Database models:** `User` (UUID primary key, bcrypt-hashed password, role enum, `isEmailVerified`, `googleId`), `RefreshToken`.

#### Other Domain Services

Each service follows the same structural pattern: `src/{config,controllers,middleware,models,routes,services,utils}/server.ts`. Each independently:
* Opens its own Sequelize connection to its private PostgreSQL database
* Validates incoming requests with Joi schemas
* Performs CRUD via Sequelize model methods
* Returns the standard `{ success, message, data }` envelope
* Exposes a `/health` endpoint and a Prometheus-compatible `/metrics` text endpoint

**Messaging Service (Port 3007)** is the architectural exception: it creates an `http.Server` wrapping the Express `Application`, then instantiates `socket.io` Server on top of that HTTP server. The Socket.IO server uses a `redis` v4 client for the pub/sub adapter, enabling horizontal scaling where multiple Messaging Service replicas can share real-time session state through Redis.

**Notification Service (Port 3008)** uses `nodemailer` 6 for transactional email (SMTP credentials from env) and `web-push` 3 (VAPID key-pair from env) for browser/mobile Web Push API notifications.

### 3.4 Data Layer

#### Primary Database — PostgreSQL 16

Each of the nine microservices owns a completely isolated PostgreSQL database. In the local development Docker Compose setup, all nine databases run within a single `postgres:16-alpine` container, provisioned at startup by the `init-multiple-databases.sh` init script:

| Service | Database Name |
|---|---|
| Auth Service | `decp_auth` |
| User Service | `decp_users` |
| Feed Service | `decp_feed` |
| Jobs Service | `decp_jobs` |
| Events Service | `decp_events` |
| Research Service | `decp_research` |
| Messaging Service | `decp_messaging` |
| Notification Service | `decp_notifications` |
| Analytics Service | `decp_analytics` |

In the production Kubernetes/AWS deployment, the Terraform `database.tf` provisions an **AWS RDS PostgreSQL 16** instance in a dedicated database subnet, accessed only from inside the EKS cluster security group. Connection pooling parameters: max 10, min 0, acquire timeout 30s, idle timeout 10s. SSL is enforced on all non-local connections (`DB_SSL=true`, `rds.force_ssl=1`).

#### ORM — Sequelize 6

All nine services use **Sequelize 6.35** with the `pg` / `pg-hstore` dialect driver. Models are defined as TypeScript classes extending `Model<T>` with `DataTypes` column definitions. Primary keys are `DataTypes.UUID` with `defaultValue: DataTypes.UUIDV4`. Schema migrations are managed via `sequelize.sync({ alter: true })` in development (`syncDb` scripts) rather than Sequelize Migrations files. Associations (e.g. `User hasMany RefreshToken`, `Post hasMany Comment`) are defined in each service's `models/index.ts`.

#### Caching / Pub-Sub — Redis 7

Redis serves two distinct roles:
1. **Socket.IO pub/sub adapter** (Messaging Service): the `redis` v4 client connects to `REDIS_URL` from env to enable Socket.IO's multi-process message fan-out. In Docker Compose, this is `redis://redis:6379`; in production it is an **AWS ElastiCache Redis 7** cluster (`redis7` parameter group family, `allkeys-lru` eviction, active defragmentation enabled).
2. **Rate limit counters** (implicit): `express-rate-limit` uses an in-memory store by default, but Redis is available for distributed deployments.

#### File Storage

In the Docker Compose local environment, uploaded files (post media, research documents) are persisted to named Docker volumes (`feed_uploads`, `research_uploads`) mounted at `/app/uploads` in the respective service containers. The Feed Service exposes these as static files via `express.static`, which the API Gateway proxies at the `/uploads/*` path.

In the production AWS deployment, the Terraform `storage.tf` provisions an **S3 bucket** (`decp-platform-assets-{env}-{account_id}`) fronted by a **CloudFront distribution**. The S3 bucket has all public access blocked; CloudFront accesses it via an Origin Access Control (OAC) policy. Server-side encryption uses **AWS KMS** (`aws:kms`). Versioning is enabled in production.

## 4. Data Flow & Request Lifecycle

### 4.1 Authenticated REST Request (Example: `GET /api/v1/jobs`)

1. **Client → API Gateway:** The React frontend (RTK Query `baseQueryWithReauth`) or mobile app (Axios instance) sends `GET https://api.decp.io/api/v1/jobs` with `Authorization: Bearer <accessToken>` header.
2. **Gateway — Security middleware chain:** Helmet sets security headers; CORS validates the origin; express-rate-limit increments the IP counter (rejects with 429 if exceeded).
3. **Gateway — correlationIdMiddleware:** Checks for an inbound `X-Correlation-ID`; if absent, generates a new UUID v4 and attaches it to both `req.headers` and the response.
4. **Gateway — authMiddleware:** Calls `jsonwebtoken.verify(token, JWT_SECRET)`. On success, the decoded `{ id, email, role, ... }` payload is written to `req.user`. On failure, returns 401.
5. **Gateway — Route match:** `router.get('/api/v1/jobs', optionalAuthMiddleware, createProxyHandler(config.services.jobs, 'jobs'))` matches.
6. **Gateway — createProxyHandler:** Strips `/api/v1/jobs` → `/`, constructs target URL `http://jobs-service:3004/`. Builds forwarded header set: `Authorization`, `x-user-id`, `x-user-role`, `x-user-email`, `x-internal-token`, `x-correlation-id`. Since the Content-Type is not multipart, delegates to `axios({ method: 'GET', url, headers, timeout: 30000 })`.
7. **Jobs Service — internalAuthMiddleware:** Validates `x-internal-token` against `process.env.INTERNAL_SERVICE_TOKEN`. Rejects with 403 if absent/invalid.
8. **Jobs Service — Controller:** `getJobs()` reads `x-user-id` from request headers. Constructs a Sequelize `findAll` query with `where: { status: 'active' }` plus pagination parameters. Returns results.
9. **Jobs Service → Gateway → Client:** Jobs Service returns `{ success: true, data: [...] }` to the Gateway's axios call. Gateway forwards `res.status(200).json(data)` back to the client. RTK Query caches the response under the `Job` tag.

### 4.2 Authentication Flow — Login

1. Client sends `POST /api/v1/auth/login` with `{ email, password }` JSON body.
2. `strictRateLimiter` checks the IP; allows max 5 attempts per 15 minutes.
3. Gateway proxies to Auth Service — no `authMiddleware` (public route).
4. Auth Service `loginController`: Joi validates body. Queries `decp_auth.users` by email. Calls `user.comparePassword(password)` (bcrypt.compare). On mismatch, returns 401.
5. On success: calls `generateTokens(userId, role)` → `jwt.sign()` produces 15-minute access token and 7-day refresh token. Stores refresh token record in `decp_auth.refresh_tokens` via `RefreshToken.create()`.
6. Returns `{ accessToken, refreshToken, user }`. Gateway forwards to client.
7. Client stores `accessToken` in Redux state; `refreshToken` in `localStorage` (web) or `expo-secure-store` (mobile).

### 4.3 Real-Time Messaging Flow (WebSocket)

1. After login, the client opens a Socket.IO WebSocket connection directly to the Messaging Service (via the NGINX Ingress `ws.decp.io` rule or the local `localhost:3007` address).
2. The Messaging Service's `setupSocketIO(io)` function authenticates the handshake by extracting the JWT from the `auth.token` Socket.IO handshake parameter and calling `jwt.verify()` with `JWT_SECRET`.
3. On successful authentication, the socket is joined to a room named by the user's ID (for private delivery).
4. Client emits `send_message` with `{ conversationId, content }`.
5. Server socket handler: persists the message via Sequelize to `decp_messaging` DB. Emits `new_message` to all room members via the Redis pub/sub adapter (ensuring delivery across multiple replica instances).
6. Recipients receive the `new_message` event on their connected socket and the Redux `socket` slice updates the UI in real time.

### 4.4 File Upload Flow (Post Media)

1. Client calls `POST /api/v1/posts/upload` with `Content-Type: multipart/form-data`.
2. API Gateway detects `multipart/form-data` content type and switches to the native `http.request` streaming pipe strategy (bypassing `axios` body buffering to avoid memory issues with large files).
3. Gateway streams the raw multipart body bytes to `http://feed-service:3003/posts/upload`, preserving the `content-type` boundary and `content-length` headers.
4. Feed Service's Multer middleware (`memoryStorage`, `limits: { fileSize: 20971520 }`) receives the stream, reads files into memory buffers.
5. `validateAndSaveFiles` middleware iterates each buffer, reads the first N bytes, and compares against the `SIGNATURES` magic-byte map (e.g., JPEG: `FF D8 FF`). Rejects files whose binary signature does not match any allowed type.
6. Valid files are written to the `uploads/` directory (Docker volume) and their paths are returned as URLs.
7. The client includes the returned file URLs in the `POST /api/v1/posts` body to create the post record.

## 5. External Integrations

* **Google OAuth 2.0 (`passport-google-oauth20` 2.0):** Provides social sign-in through the Auth Service. Requires `GOOGLE_CLIENT_ID` and `GOOGLE_CLIENT_SECRET` environment variables. On first login, creates a user record with `googleId` populated and `isEmailVerified: true`. Google is used solely for identity; no Google APIs are called after the initial token exchange.

* **SMTP Email Provider (Nodemailer 6.9):** Used by both the Auth Service (email verification, password reset) and the Notification Service (transactional notifications). Configured via `SMTP_HOST`, `SMTP_PORT`, `SMTP_USER`, `SMTP_PASS` environment variables, making it compatible with any SMTP provider (SendGrid, AWS SES, Mailgun, etc.). The `from` address is `noreply@decp.edu`. [Requires Manual Verification: Confirm which SMTP provider is used in production.]

* **Web Push API / VAPID (`web-push` 3.6):** Used by the Notification Service to send browser and mobile push notifications without a third-party push service. VAPID public/private key pair configured via `VAPID_PUBLIC_KEY` and `VAPID_PRIVATE_KEY` environment variables. Push subscriptions are stored in the `decp_notifications` database.

* **AWS S3 (`aws_s3_bucket` — Terraform):** Stores user-uploaded media (post images/videos) and research documents in production. Accessed from application services via the AWS SDK. The bucket is private; asset delivery is through CloudFront with Origin Access Control.

* **AWS CloudFront (`aws_cloudfront_distribution` — Terraform):** CDN for static asset delivery from S3 (KMS-encrypted origin). Also used to serve the compiled React SPA built artefact. TLS is enforced. Deployed in `us-east-1` (the CloudFront-required region).

* **AWS EKS (Kubernetes) (`module "eks"` — Terraform implied by `EKS_CLUSTER_NAME`):** Production container orchestration. Services run as K8s Deployments (`replicas: 3` for the API Gateway) with `RollingUpdate` strategy. Resource requests/limits are defined per pod (API Gateway: 250m CPU / 256Mi memory requests; 1000m CPU / 1Gi memory limits).

* **AWS RDS PostgreSQL 16 (`aws_db_instance` — `database.tf`):** Managed database in production. PostgreSQL 16 family parameter group; connection logging enabled; SSL enforced; deployed in database subnets, accessible only from EKS node security group.

* **AWS ElastiCache Redis 7 (`aws_elasticache_replication_group` implied by `cache.tf`):** Managed Redis in production for Socket.IO pub/sub. Parameter group `redis7`; `allkeys-lru` eviction; TCP keepalive 300s; deployed in private subnets.

* **GitHub Container Registry (GHCR) (`ghcr.io` — CI/CD):** Docker image registry. All services are built and pushed to `ghcr.io/decp-platform/{service-name}:sha` tags via the `CD - Backend Services` GitHub Actions workflow using `aws-actions/configure-aws-credentials` and docker buildx.

* **Codecov (`codecov/codecov-action@v3` — `ci.yml`):** Automated test coverage reporting. Coverage LCOV files from each service's Jest tests are uploaded per-service with flags.

* **GitHub CodeQL (`github/codeql-action` — `security-scan.yml`):** Static Application Security Testing (SAST) for JavaScript/TypeScript. Runs on push to `main`/`develop`, on pull requests, and on a daily schedule at 02:00 UTC. Uses `security-extended` and `security-and-quality` query suites.

## 6. Infrastructure & Deployment

### Containerization

Every service (API Gateway, all 9 microservices, frontend) has a multi-stage or single-stage **Dockerfile**:

* **Backend services:** `FROM node:20-alpine`. Production images (`Dockerfile`) copy only the compiled `dist/` output and install `--only=production` dependencies. Development images (`Dockerfile.dev`) install all dependencies and run `ts-node-dev` for hot-reload. A `HEALTHCHECK` using `wget --spider http://localhost:{PORT}/health` is configured with 30-second intervals.
* **Frontend:** Two-stage build. Stage 1 `FROM node:20-alpine` runs `npm ci && tsc && vite build`. Stage 2 `FROM nginx:alpine` copies the `dist/` output and a custom `nginx.conf` into the NGINX document root. Exposes port 80.
* **Mobile:** A `Dockerfile` exists for the React Native project (Expo for web target or OTA update serving), though native iOS/Android builds are typically produced via `expo build` or EAS Build rather than Docker.

**Local development** uses `docker compose up` from `backend/docker-compose.yml`:
* Single PostgreSQL 16 container with the `init-multiple-databases.sh` init script creating all 9 databases
* Single Redis 7 container
* All 9 microservices + API Gateway in individual containers on the `decp-network` bridge network
* Named volumes `postgres_data`, `feed_uploads`, `research_uploads` for persistence

### Kubernetes Configuration

Production K8s manifests are in `infrastructure/k8s/`:

* **Namespace:** `decp-platform`
* **Ingress:** NGINX Ingress Controller with `cert-manager` for TLS termination. SSL redirect enforced. Rate limiting: 10 RPS / 5 connections per IP at the ingress layer (in addition to the Gateway's application-layer rate limits). Three `host` rules: `api.decp.io` → API Gateway service `:3000`; `app.decp.io` → frontend service `:3000`; `ws.decp.io` → WebSocket endpoint.
* **Deployments:** Each service deployed as a K8s `Deployment` with `replicas: 3` (API Gateway confirmed; others follow the same template), `RollingUpdate` strategy (maxSurge 25%, maxUnavailable 25%), non-root security context (`runAsUser: 1001`), Prometheus scrape annotations.
* **Secrets:** Managed as K8s `Secret` objects (`database-secrets`, `auth-secrets`, `rabbitmq-secrets`) injected as environment variables via `valueFrom.secretKeyRef`.
* **RBAC:** `rbac.yml` defines `ServiceAccount`, `ClusterRole`, and `ClusterRoleBinding` for IRSA-based AWS resource access.
* **PVCs:** `pvc.yml` defines `PersistentVolumeClaim` objects for upload volumes (backed by AWS EBS in production via the EBS CSI driver).

### Observability

* **Prometheus:** Scrapes all service pods at `:9090/metrics` (15-second interval) using Kubernetes SD config. Services expose a custom Prometheus text-format `/metrics` endpoint built without an external library (measured from `process.memoryUsage()` and `process.uptime()`).
* **Grafana:** Connected to both Prometheus (metrics) and Loki (logs) as data sources. Custom dashboards stored under `infrastructure/monitoring/grafana/`.
* **Loki:** Log aggregation. Services log to stdout via Winston; Loki collects and indexes them for query in Grafana.
* **Alertmanager:** Alert routing configuration in `alertmanager.yml`; alert rule definitions in `alert-rules.yml`. Connected to Prometheus.
