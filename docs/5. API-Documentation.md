## 1. Overview

The DECP Platform exposes a RESTful HTTP API through a single **API Gateway** that proxies requests to internally-isolated microservices. All public-facing interaction occurs through this gateway; microservices are not directly reachable from outside the cluster.

| Property | Value |
|----------|-------|
| **Production Base URL** | `https://api.decp.io/api/v1` |
| **Local Development URL** | `http://localhost:3000/api/v1` |
| **Default Content-Type** | `application/json` |
| **File Upload Content-Type** | `multipart/form-data` |
| **API Version** | v1 |

### Response Envelope

All endpoints return responses wrapped in a consistent JSON envelope:

```json
{
  "success": true,
  "message": "Human-readable description",
  "data": { ... }
}
```

Error responses:

```json
{
  "success": false,
  "message": "Error description",
  "error": "Detailed error message or validation info"
}
```

Paginated endpoints additionally include a `meta` object:

```json
{
  "success": true,
  "data": [ ... ],
  "meta": {
    "page": 1,
    "limit": 10,
    "total": 100,
    "totalPages": 10
  }
}
```

## 2. Authentication

The platform uses **stateless JWT Bearer Token** authentication with a two-token scheme.

### Token Types

| Token | Expiry | Purpose |
|-------|--------|---------|
| **Access Token** | 15 minutes | Sent with every authenticated API request |
| **Refresh Token** | 7 days | Used to obtain a new access token; stored in the database and revocable |

### Making Authenticated Requests

Include the access token in the `Authorization` header of every protected request:

```http
Authorization: Bearer <accessToken>
```

### Token Refresh Flow

When an access token expires, use the refresh token to obtain a new token pair:

```http
POST /api/v1/auth/refresh
Content-Type: application/json

{ "refreshToken": "<refreshToken>" }
```

A new `accessToken` and `refreshToken` pair will be returned. Replace both stored tokens.

### Auth Requirement Key

Throughout this document, the **Auth Required** column uses these levels:

| Level | Meaning |
|-------|---------|
| **Required** | `Authorization: Bearer <accessToken>` must be present and valid |
| **Optional** | Request succeeds with or without a token; token unlocks additional behaviour |
| **None** | No authentication required |

## 3. Error Handling

### HTTP Status Codes

| Code | Meaning |
|------|---------|
| `200` | OK — request succeeded |
| `201` | Created — resource created successfully |
| `204` | No Content — success with no response body |
| `400` | Bad Request — validation error or malformed input |
| `401` | Unauthorized — missing or invalid auth token |
| `403` | Forbidden — authenticated but not permitted for this action |
| `404` | Not Found — resource does not exist |
| `409` | Conflict — resource already exists (e.g. duplicate email, already applied) |
| `500` | Internal Server Error — unexpected server-side failure |

### Common Validation Error (400)

```json
{
  "success": false,
  "message": "Validation error",
  "error": "\"email\" must be a valid email address"
}
```

## 4. Rate Limiting

| Rule | Limit | Applies To |
|------|-------|------------|
| **Global** | 100 requests per 15 minutes per IP | All endpoints |
| **Strict** | 5 requests per 15 minutes per IP | `/auth/register`, `/auth/login`, `/auth/forgot-password`, `/auth/reset-password` |

When the limit is exceeded the gateway returns HTTP `429 Too Many Requests`.

## 6. WebSocket Events

Real-time functionality (live messaging, online presence, instant notifications) is provided via a Socket.IO server.

| Property | Value |
|----------|-------|
| **Production WebSocket URL** | `wss://ws.decp.io` |
| **Local Development URL** | `ws://localhost:3007` |
| **Protocol** | Socket.IO over WebSocket |

### Connection & Authentication

Connect with a JWT access token in the Socket.IO handshake:

```javascript
import { io } from "socket.io-client";

const socket = io("wss://ws.decp.io", {
  auth: { token: "<accessToken>" }
});
```

### Client → Server Events

| Event | Payload | Description |
|-------|---------|-------------|
| `join_conversation` | `{ conversationId: "uuid" }` | Subscribe to a conversation room |
| `leave_conversation` | `{ conversationId: "uuid" }` | Unsubscribe from a conversation room |
| `send_message` | `{ conversationId: "uuid", content: "string", type: "text" }` | Send a real-time message |
| `typing` | `{ conversationId: "uuid" }` | Indicate the user is typing |

### Server → Client Events

| Event | Payload | Description |
|-------|---------|-------------|
| `new_message` | `{ ...message }` | A new message was sent in a conversation |
| `user_typing` | `{ userId: "uuid", conversationId: "uuid" }` | Another user is typing |
| `notification` | `{ ...notification }` | A new notification for the authenticated user |
| `user_online` | `{ userId: "uuid" }` | A connected user came online |
| `user_offline` | `{ userId: "uuid" }` | A connected user went offline |
