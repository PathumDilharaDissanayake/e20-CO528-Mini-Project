## 1. Identified User Roles (Actors)

The following roles are defined in the platform's role-based access control system. They are enforced at the backend via the `x-user-role` header injected by the API Gateway after JWT verification, and at the frontend via `user.role` from the Redux auth store.

| Actor ID | Role | Description | Source |
|----------|------|-------------|--------|
| **A1** | **Public Guest** | Unauthenticated visitor. Can browse the public feed, job listings, events, and research projects. Cannot create, react to, comment on, or otherwise interact with content. | `optionalAuthMiddleware` in `api-gateway/src/middleware/authMiddleware.ts` |
| **A2** | **Student** | Authenticated user with `role = 'student'`. Full read-write access to feed, profile, connections, job applications, event RSVPs, and research collaboration. Cannot post jobs. | `registerSchema` role enum; `x-user-role` checks in controllers |
| **A3** | **Alumni** | Authenticated user with `role = 'alumni'`. All student capabilities plus the ability to **create job postings**. | `jobController.ts` — allows `['alumni', 'faculty', 'admin']` |
| **A4** | **Faculty** | Authenticated user with `role = 'faculty'`. All student capabilities plus the ability to **create job postings**, create events, and create research projects. Graduation year is optional during registration. | `jobController.ts`; `eventController.ts`; `researchController.ts` |
| **A5** | **Admin** | Authenticated user with `role = 'admin'`. Unrestricted platform access — can delete any content, delete any user, update application statuses on any job, and access the analytics dashboard. | Role check pattern `userRole !== 'admin'` in multiple controllers |

## Epic 1 — User Account Management

**Goal:** Enable visitors to register, authenticate, recover access, and manage their session securely.

---

### US-1.1 — Register a New Account

**User Story:**  
> As a **Public Guest**, I want to create a new account using my academic email, personal details, and role, so that I can join the DECP community.

**Primary Use Case Flow:**
1. Guest navigates to `/register`.
2. **Step 1 (Account):** Enters email address, password, and password confirmation.
3. **Step 2 (Personal):** Enters first name, last name, and selects platform role (`student`, `alumni`, or `faculty`).
4. **Step 3 (Academic):** Selects department and, if role is `student` or `alumni`, enters graduation year.
5. Guest submits the form.
6. System validates all fields and calls `POST /api/auth/register`.
7. On success, the JWT access token and refresh token are stored; Redux `setCredentials` is dispatched; user is redirected to `/`.

**Acceptance Criteria (from `registerSchema` in `frontend/src/utils/validators.ts` and auth-service Joi schemas):**
- [ ] Email must be a valid email address format.
- [ ] Password must be at least **6 characters** long.
- [ ] `confirmPassword` must exactly match `password`; otherwise display "Passwords do not match".
- [ ] `firstName` must be at least **2 characters** long.
- [ ] `lastName` must be at least **2 characters** long.
- [ ] `role` must be one of: `student`, `alumni`, `faculty`.
- [ ] `department` is required (min 1 character).
- [ ] `graduationYear` is required when `role` is `student` or `alumni`; optional for `faculty`.
- [ ] On field validation error, the error message is displayed inline per field.
- [ ] On successful registration, user is authenticated and redirected to the feed.
- [ ] On duplicate email, system returns a 409 error and displays the message to the user.
- [ ] Registration rate-limited to **5 requests per 15 minutes** per IP (`strictRateLimiter`).

**Technical Mapping:**
| Layer | File / Endpoint |
|-------|----------------|
| Frontend: Page | `frontend/src/pages/RegisterPage.tsx` |
| Frontend: Validation | `frontend/src/utils/validators.ts` → `registerSchema` |
| Frontend: Constants | `frontend/src/utils/constants.ts` → `ROLES`, `DEPARTMENTS` |
| Frontend: State | `frontend/src/features/authSlice.ts` → `setCredentials` |
| Backend: Service | `auth-service` |
| API Route | `POST /api/auth/register` |
| Rate Limiter | `strictRateLimiter` (5 req / 15 min) |

### US-1.2 — Log In to Existing Account

**User Story:**  
> As a **Student**, **Alumni**, **Faculty**, or **Admin**, I want to log in with my email and password, so that I can access authenticated platform features.

**Primary Use Case Flow:**
1. User navigates to `/login`.
2. Email field is auto-focused on page mount.
3. User enters email and password and submits the form.
4. System validates using `loginSchema` and calls `POST /api/auth/login`.
5. On success: JWT tokens received; `setCredentials` dispatched; user redirected to `?redirect=` path or `/` by default.
6. On failure: error displayed for 5 seconds then auto-cleared.

**Acceptance Criteria (from `loginSchema` in `frontend/src/utils/validators.ts`):**
- [ ] Email must be a valid email address format.
- [ ] Password must be at least **6 characters** long.
- [ ] Email field is auto-focused on component mount.
- [ ] Error messages are automatically cleared after **5 seconds**.
- [ ] On successful login, user is redirected to the `?redirect=` query parameter path if present, otherwise to `/`.
- [ ] On invalid credentials, a 401 error message is displayed.
- [ ] Rate-limited to **5 requests per 15 minutes** per IP.

**Technical Mapping:**
| Layer | File / Endpoint |
|-------|----------------|
| Frontend: Page | `frontend/src/pages/LoginPage.tsx` |
| Frontend: Validation | `frontend/src/utils/validators.ts` → `loginSchema` |
| Backend: Service | `auth-service` |
| API Route | `POST /api/auth/login` |

### US-1.3 — Request Password Reset

**User Story:**  
> As any **authenticated or unauthenticated user**, I want to reset my forgotten password by providing my email address, so that I can recover access to my account.

**Primary Use Case Flow:**
1. User navigates to `/forgot-password`.
2. User enters their registered email address.
3. System validates using `forgotPasswordSchema` and calls `POST /api/auth/forgot-password`.
4. On submission, the page transitions to a "Check Your Email" confirmation screen regardless of whether the email exists (prevents email enumeration).
5. User receives an email with a reset link containing a time-limited token.
6. User clicks the link, navigates to the reset page `/reset-password?token=<token>`.
7. User enters and confirms a new password.
8. System calls `POST /api/auth/reset-password` with the token and new password.

**Acceptance Criteria (from `forgotPasswordSchema` and `resetPasswordSchema` in `frontend/src/utils/validators.ts`):**
- [ ] Email must be a valid email address format.
- [ ] After form submission, page transitions to a "Check Your Email" success screen.
- [ ] The confirmation screen is always shown regardless of whether the email is registered (to prevent email enumeration attacks).
- [ ] For reset: new password must be at least **6 characters**.
- [ ] For reset: confirm password must match the new password.
- [ ] Reset token must be valid and unexpired; expired tokens return a 400 error.

**Technical Mapping:**
| Layer | File / Endpoint |
|-------|----------------|
| Frontend: Forgot Password | `frontend/src/pages/ForgotPasswordPage.tsx` |
| Frontend: Reset Password | `frontend/src/pages/ResetPasswordPage.tsx` (inferred) |
| Frontend: Validation | `frontend/src/utils/validators.ts` → `forgotPasswordSchema`, `resetPasswordSchema` |
| API Route: Request | `POST /api/auth/forgot-password` |
| API Route: Reset | `POST /api/auth/reset-password` |

**Technical Mapping:**

### US-1.5 — Refresh Authentication Token

**User Story:**  
> As an **authenticated user**, I want my session to remain active automatically, so that I am not interrupted by unexpected logouts during extended use.

**Primary Use Case Flow:**
1. Frontend detects that the access token has expired (or will expire imminently).
2. System calls `POST /api/auth/refresh` using the stored refresh token (sent via HTTP-only cookie or request body).
3. A new access token (15-minute expiry) is returned.
4. Frontend updates the stored token and retries the original request.

**Acceptance Criteria:**
- [ ] Access token lifespan: **15 minutes**.
- [ ] Refresh token lifespan: **7 days**.
- [ ] Invalid or expired refresh token returns a 401 error and triggers logout.
- [ ] Users are not interrupted if the access token is refreshed transparently.

**Technical Mapping:**
| Layer | File / Endpoint |
|-------|----------------|
| Frontend: RTK Query | `frontend/src/services/authApi.ts` |
| Backend: Service | `auth-service` |
| API Route | `POST /api/auth/refresh` |

### US-1.6 — Log Out of Account

**User Story:**  
> As an **authenticated user**, I want to log out of the platform, so that my session is terminated and my account is secured.

**Primary Use Case Flow:**
1. User clicks the "Logout" button in the navigation or settings.
2. System calls `POST /api/auth/logout`.
3. Server invalidates the refresh token.
4. Frontend clears the Redux auth state and stored tokens, then redirects to `/login`.

**Acceptance Criteria:**
- [ ] After logout, all authenticated API requests return 401 until the user logs in again.
- [ ] User is redirected to `/login` after logout.
- [ ] Refresh token is invalidated on the server side.

**Technical Mapping:**
| Layer | File / Endpoint |
|-------|----------------|
| Frontend: Auth Slice | `frontend/src/features/authSlice.ts` → `logout` action |
| API Route | `POST /api/auth/logout` |

## Epic 3 — Social Connections & Networking

**Goal:** Enable users to discover and connect with peers, alumni, and faculty through a mutual connection system.

---

### US-3.1 — Send a Connection Request

**User Story:**  
> As an **authenticated user**, I want to send a connection request to another user, so that I can grow my professional network.

**Primary Use Case Flow:**
1. User navigates to another user's profile or views a suggested user card on the feed.
2. User clicks "Connect".
3. System calls `POST /api/users/connections/:id/follow`.
4. The button state changes to "Pending".

**Acceptance Criteria:**
- [ ] "Connect" button is visible only when the connection status is `none`.
- [ ] After clicking, button updates to "Pending" immediately (optimistic update).
- [ ] Target user does not appear in "My Connections" until they accept.
- [ ] A notification is sent to the target user.

**Technical Mapping:**
| Layer | File / Endpoint |

### US-3.2 — Accept or Decline a Connection Request

**User Story:**  
> As an **authenticated user**, I want to accept or decline incoming connection requests, so that I can control who is in my professional network.

**Primary Use Case Flow:**
1. User's own profile page polls `GET /api/users/connections/requests` every **30 seconds** to display pending incoming requests.
2. For each pending request, the user sees "Accept" and "Decline" buttons.
3. Accept: system calls `PUT /api/users/connections/:id/accept`.
4. Decline: system calls `DELETE /api/users/connections/:id/decline`.
5. The request is removed from the pending list.

**Acceptance Criteria:**
- [ ] Pending connection requests are polled every **30 seconds** on own profile page.
- [ ] Accepting a request adds both users to each other's connection list.
- [ ] Declining removes the request without adding a connection.
- [ ] Connection count on the profile updates after acceptance.

**Technical Mapping:**
| Layer | File / Endpoint |
|-------|----------------|
| Frontend: Page | `frontend/src/pages/ProfilePage.tsx` |
| Frontend: API | `useAcceptConnectionMutation`, `useDeclineConnectionMutation` in `userApi.ts` |
| API Routes | `PUT /api/users/connections/:id/accept`, `DELETE /api/users/connections/:id/decline` |

- [ ] SuggestedUsersWidget shows recommended connections for authenticated users.

**Technical Mapping:**
| Layer | File / Endpoint |

### US-4.2 — Create a Text Post

**User Story:**  
> As an **authenticated user**, I want to create a text post visible in the feed, so that I can share updates and ideas with the community.

### US-4.3 — Create a Post with Media (Image/Video/Document)

**User Story:**  
> As an **authenticated user**, I want to attach images, videos, or documents to my posts, so that I can share richer content.

**Primary Use Case Flow:**
1. In the `CreatePost` composer, user clicks an attachment button (image, video, or document).
2. User selects a file(s) from their device.
3. System uploads the file via `POST /api/posts/upload` (multipart/form-data).
4. Uploaded file URL is attached to the post payload.
5. User clicks "Post"; system calls `POST /api/posts` with `type: 'image'|'video'|'document'` and the media URL.

3. For **direct** conversations (`type: 'direct'`): one participant is selected; if a conversation already exists, the existing one is returned (deduplication).
4. For **group** conversations (`type: 'group'`): multiple participants are selected and a group title is entered.
5. System calls `POST /api/messaging/conversations`.

**Acceptance Criteria (from messaging-service Joi `createConversation` schema):**
- [ ] `type` must be `direct` or `group`.
- [ ] `title` is required for group conversations (max **200 characters**); optional for direct.
- [ ] `participants` must be an array with at least **1 valid UUID**.
- [ ] For direct conversations, if a conversation with those two participants already exists, it is returned without creating a duplicate.

**Technical Mapping:**
| Layer | File / Endpoint |
|-------|----------------|
| Frontend: Page | `frontend/src/pages/MessagesPage.tsx` |
| Frontend: Component | `frontend/src/components/messaging/ChatList.tsx` |
| API Route | `POST /api/messaging/conversations` |

### US-8.3 — Send and Receive Messages in Real Time

**User Story:**  
> As an **authenticated user** in a conversation, I want to send and receive messages in real time, so that my conversations feel instant and responsive.

**Primary Use Case Flow:**
1. User opens a conversation; `ChatRoom` initialises a Socket.IO connection and subscribes to the conversation room.
2. User types a message (1–2,000 characters) and sends.
3. System persists the message via `POST /api/messaging/conversations/:id/messages`.
4. Message is simultaneously broadcast via Socket.IO to all conversation participants.
5. Other participants' `ChatRoom` instances receive the event and append the message in real time without polling.

**Acceptance Criteria (from `messageSchema` in `frontend/src/utils/validators.ts` and messaging-service Joi):**
- [ ] Message `content` must be between **1 and 2,000 characters** (`messageSchema.content.min(1).max(2000)`).
- [ ] Message `type` must be one of: `text`, `image`, `file` (default: `text`).
- [ ] Messages are delivered in real time via Socket.IO without requiring a page refresh.
- [ ] Message history is loaded on conversation open via `GET /api/messaging/conversations/:id/messages`.

**Technical Mapping:**
| Layer | File / Endpoint |
|-------|----------------|
| Frontend: Page | `frontend/src/pages/MessagesPage.tsx` |
| Frontend: Component | `frontend/src/components/messaging/ChatRoom.tsx` |
